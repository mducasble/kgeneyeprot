The app already records IMU data to `imu.jsonl`, but the current implementation has synchronization issues:
- accelerometer and gyroscope samples are being grouped together even when their timestamps do not match closely enough
- some early gyro samples are zeroed
- multiple output samples may share the same top-level timestamp even though the internal accel/gyro timestamps differ
- this creates pseudo-sync instead of true sample alignment

Your task is to PATCH the current IMU pipeline to produce cleaner, better synchronized IMU output.

IMPORTANT:
- Do NOT rewrite the app
- Do NOT change the recording flow
- Do NOT change metadata/qc/upload logic except where needed to support the IMU fix
- Only improve the IMU capture and serialization quality

Goal:

Tasks:

- if available, use DeviceMotion as the primary source for IMU output
- extract:
  - acceleration

- map them into the existing IMU output schema:
{
  "timestampEpochMs": number,
  "relativeMs": number,
  "accel": { "x": number, "y": number, "z": number },
  "gyro": { "x": number, "y": number, "z": number }
}

Requirements:

2) FALLBACK: SYNCHRONIZED MERGING OF ACCEL + GYRO
==================================================

If separate Accelerometer and Gyroscope streams must still be used:

Implement a proper synchronization buffer.

Create:
- a rolling buffer for accelerometer samples
- a rolling buffer for gyroscope samples

For each candidate output sample:
- match accel and gyro samples by nearest timestamp
- only emit an IMU row if:
  abs(accel.timestamp - gyro.timestamp) <= 5ms

Rules:
- do NOT blindly pair the latest accel with the latest gyro
- do NOT emit samples where gyro is still zero/uninitialized unless that is truly the measured value during steady state
- discard unmatched stale samples outside a reasonable window
- prefer temporal consistency over raw sample count

3) FIX TOP-LEVEL TIMESTAMP SEMANTICS
==================================================

Current issue:
- top-level timestampEpochMs may not represent a true synchronized sample instant

Goal:
- ensure each emitted IMU row has one meaningful timestamp

Rules:
- when accel and gyro are merged, set:
  timestampEpochMs = round((accel.timestampEpochMs + gyro.timestampEpochMs) / 2)
- set:
  relativeMs = timestampEpochMs - imuStartEpochMs
- do NOT emit repeated top-level timestamps unless they are genuinely produced by the source stream

==================================================

Current issue:

Goal:

Tasks:

- do not start persisting IMU rows until both motion sources are producing non-empty, timestamped data consistently
- warmup can be either:
  - fixed duration (e.g. 100–300 ms)
  - or readiness-based (preferred)

Readiness-based rule:
- begin writing only after both accel and gyro have produced at least N valid samples with close timestamp alignment

Do NOT change the external session start semantics.
If needed, keep `imuStartEpochMs` as the start request time, but add a metadata field for:
